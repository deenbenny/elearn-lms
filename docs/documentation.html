Community Skill Exchange Platform  ·  Full-Stack MERN Application  ·  v1.0

📋 Project Overview

ELearnLMS is a credit-based skill exchange marketplace where users can offer skills they know and learn skills they want — entirely through a virtual credit economy. Learners spend credits to attend sessions; instructors earn credits by teaching. No real money changes hands.

🏗️ Architecture

🔄 Session Workflow

The session lifecycle follows a strict approval-based flow with credit deductions triggered at specific state transitions.

State Machine

PENDINGCreated by learner
→
APPROVEDCredits deducted
→
COMPLETEDInstructor earns
PENDING
→
REJECTEDNo credits move
PENDING
→
CANCELLEDBy learner

Credit Economy

EventLearner CreditsInstructor CreditsNotes
Registration+5 starter+5 starterBoth roles get initial credits
Session requestedNo changeNo changeCredits reserved mentally, not deducted
Session approved-creditsUsedNo changeDeducted from learner immediately
Session completedNo change+creditsUsedAlso increments instructor.totalSessions
Session rejectedNo changeNo changeNothing moved
Session cancelledNo changeNo changeOnly cancellable before approval

📌 Assumptions & Design Decisions

  • 1
    Virtual credit economy only. No real payment gateway is integrated. The credit system is a virtual economy used purely for gamification and session exchange. Credits cannot be purchased with money.
  • 2
    Credit deduction happens at approval, not at booking. This gives instructors the ability to reject unwanted requests before any credits change hands, preventing runaway deductions.
  • 3
    A single user cannot be both learner and instructor simultaneously in the same session. The session model separately tracks learner and instructor ObjectId references, and the booking flow prevents the skill's own instructor from requesting their own skill.
  • 4
    Time-slot conflict is checked at the skill level, not the instructor level. If the same skill is booked at the same datetime by two different learners, the second request is rejected with a 400 error. However, an instructor can teach two different skills at the same time.
  • 5
    Instructor rating is a simple arithmetic mean. It is recalculated live every time a new feedback is submitted, using all feedback ratings across all the instructor's completed sessions. No weighted rating or decay factor is applied.
  • 6
    Only the learner can submit feedback, and only once per session. If session.feedback.rating is already set, re-submission is blocked. Instructors cannot rate learners.
  • 7
    JWT tokens are stored in browser localStorage. This is a trade-off: it simplifies the auth flow and removes the need for cookie configuration, but makes the app vulnerable to XSS. This is acceptable for an academic project but should use HttpOnly cookies in production.
  • 8
    Passwords are hashed via bcrypt with the default salt rounds (10). The hashing occurs in a Mongoose pre-save hook, so the plain-text password is never stored or returned by the API.
  • 9
    Skill isActive flag is a soft toggle, not a hard delete. Admins and instructors can deactivate a skill without removing it from the database. Learners cannot request sessions for inactive skills.
  • 10
    The Admin role is assigned manually (e.g., via MongoDB Atlas or seeding). There is no in-app admin promotion UI. The register API defaults role to learner, and the admin role must be set directly in the database.
  • 11
    Session scheduling has no time-zone normalization. scheduledAt is stored as a plain ISO date string. The frontend uses an HTML datetime-local input, so times are local to the user's browser. Learner and instructor may see different local times.
  • 12
    Search uses MongoDB regex, not a full-text search engine. The skill search matches title, description, and tags using a case-insensitive regex. This is sufficient for small datasets but does not support typo-tolerance or relevance ranking. MongoDB Atlas Search or Elasticsearch would be needed for production scale.
  • 13
    The platform is single-tenant. There is no concept of organizations, schools, or isolated groups. All users see all active skills and can request sessions from any instructor.
  • 14
    No email verification or password reset flow is implemented. User accounts are activated immediately upon registration. This is a deliberate scope reduction for the academic assignment.
  • 15
    File uploads (profile pictures, session recordings) are out of scope. The platform is text-only. The recorder/ utility in the repo is a Playwright-based demo recording tool for the assignment submission, not a product feature.
